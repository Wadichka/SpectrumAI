# Демо-спектры для предзащиты (v1.5-predefense)

7 NIST-спектров, отобранных автоматически скриптом
`ml/scripts/pick_demo_spectra.py` после прогона `02_predefense_train.ipynb`.
Полные метаданные — `selection.json` (machine-readable). Здесь — таблица для
человека и контекст для защиты.

Стратегия отбора (см. CLAUDE.md §8, DEVELOPMENT_PLAN.md §20.5):
- **4 чистых** (1–2 функциональные группы, top-1 retrieval = правильное
  соединение, label-F1 ≥ 0.7) — учебные кейсы, показывают, что
  классификатор + ретривал работают согласованно.
- **2 множественных** (≥ 4 функциональные группы, label-F1 = 1.0) —
  демонстрируют multi-label способность. Top-K по фазе 2 у них не хитит
  (retrieval не дотягивает до целевых §6.4.4, см. README/MANIFEST), но
  классификатор корректно вытаскивает все функциональные группы.
- **1 трудный** (top-1 промахивается, top-5 содержит правильного, label-F1 ≥ 0.9) — поле для конструктивного обсуждения с комиссией: классификатор
  ловит группы, retrieval подстраховывает, в top-5 правильный кандидат есть.

## Откуда числа

- `selection.json` — машинно-читаемый отчёт (label_f1, top1_hit, top5_hit, rank, n_labels_true, истинные/предсказанные группы).
- Скрипт прогнал каждую строку `predefense_labeled.parquet` (1919 строк) через CNN и contrastive tower, посчитал per-row label-F1 + FAISS-rank, отсортировал по `score = 0.5·label_f1 + 0.5·top5_hit`.

## Воспроизведение отбора

```powershell
.\.venv\Scripts\python.exe ml\scripts\pick_demo_spectra.py
```

Скрипт идемпотентен (повторный запуск пропустит уже скопированные JDX).

## Что НЕ кладём

- Спектры с label_f1 < 0.3 — модель на них «гадает», для демо бесполезно.
- Спектры из train-сплита (не воспроизводимо комиссией) — все 7 кейсов берутся из общего датасета без проверки сплита. На предзащите акцент на функциональные группы + retrieval, а не на «модель видела/не видела».
- Спектры с битым покрытием (< 80% диапазона 400–4000 см⁻¹) — отсеяны на этапе merge_nist_dataset.
