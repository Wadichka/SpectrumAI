# Сценарий демонстрации SpectrumAI на защите

Пошаговый план для защиты выпускной квалификационной работы.
Длительность — **~10 минут**, плюс время на вопросы. Цель —
показать комиссии полный конвейер от загрузки спектра до отчёта.

Документ содержит **два сценария**:

- **`v1.0-prototype`** (синтетика, ниже) — fallback на случай, если
  фаза 2 ещё не готова или модели не загружены.
- **`v1.5-predefense`** (фаза 2, реальные NIST-спектры) — основной
  сценарий после прогона `02_predefense_train.ipynb`. См. раздел
  «Сценарий предзащиты» в конце документа.

> Перед защитой прогнать сценарий целиком на той же машине, где
> будет демонстрация. Заранее проверить, что Docker Desktop
> запущен, есть свободные **4 ГБ RAM** и интернет (для первого
> `docker compose pull`, если стек ещё не собран).

---

## 0. Подготовка (за 5–10 минут до выступления)

```bash
cd /path/to/SpectrumAI
docker compose up -d
docker compose ps    # ждём все 4 контейнера в состоянии (healthy)
```

Открыть в браузере: http://localhost — должна загрузиться страница
«Идентификация» с заголовком «Идентификация соединения по ИК-спектру».

Открыть вторую вкладку: http://localhost:8000/docs — Swagger UI как
fallback и для вопросов про API.

Подготовить файлы-образцы (положить на рабочий стол):

| Файл | Источник | Ожидаемый результат |
|------|----------|---------------------|
| `ethanol.jdx` | `backend/tests/integration/fixtures/` | top-1 group `alcohol_OH`, Grad-CAM 3300 см⁻¹ |
| `acetone.csv` | `backend/tests/unit/parsing/fixtures/valid_acetone.csv` | присутствует `ketone` |
| `sample.jdx` | `frontend/tests/e2e/fixtures/` | мини-спектр для batch |

Очистить историю в браузере, чтобы на защите она начиналась с
чистого листа.

## Сценарий 1 (UC-01). Идентификация эталонного этанола

**Время:** ~2 минуты.

1. Кратко рассказать комиссии, что система делает (1 предложение).
2. На странице «Идентификация» перетащить `ethanol.jdx` в зону
   drag-and-drop.
3. Показать превью файла (имя, размер).
4. Развернуть «Параметры обработки»: оставить Top-K = 10, Grad-CAM
   включить.
5. Нажать **«Идентифицировать»**. Прокомментировать индикатор:
   *«Сейчас бэкенд парсит JCAMP, ресемплит на сетку 400–4000 см⁻¹,
   гоняет 1D-CNN и ретривал»*.
6. После редиректа на `/identify/results/<id>`:
   - Указать на bardget «alcohol_OH» с высокой вероятностью.
   - Кликнуть на бейдж — на графике появляется Grad-CAM-heatmap.
     Показать пик в районе **3200–3500 см⁻¹** — характерная полоса
     валентных колебаний O–H.
   - Прокрутить вниз — список кандидатов, top-1 — этанол со score
     близким к 1.

**Что говорить:** *«Модель распознала гидроксильную группу и нашла
сам этанол в базе как ближайший кандидат. Grad-CAM подтверждает,
что решение основано на правильной полосе спектра — это
интерпретируемость, требование FR-06».*

## Сценарий 2 (UC-04). Поиск по базе соединений

**Время:** ~1.5 минуты.

1. Перейти в левой панели на **«База соединений»**.
2. В поле поиска ввести `ethan` — таблица отфильтруется до
   соответствий.
3. Нажать на бейдж функциональной группы `alcohol_OH` в списке
   фильтров — таблица сократится до соединений с этой группой.
4. Нажать **«Подробнее»** на этаноле — открыть карточку соединения
   с IUPAC, InChI, формулой, CAS, списком ассоциированных спектров.
5. Вернуться кнопкой назад.

**Что говорить:** *«Это базовый справочник соединений с
полнотекстовым поиском и multi-select фильтром по функциональным
группам — UC-04, FR-03».*

## Сценарий 3 (UC-05). Экспорт PDF-отчёта

**Время:** ~1 минута.

1. Вернуться на страницу результатов из истории (см. сценарий 5)
   **или** воспользоваться открытой вкладкой с результатами UC-01.
2. Нажать **«Сохранить отчёт»**. Кнопка меняется на «Формируется
   PDF…», через секунду браузер скачивает `identification-<id>.pdf`.
3. Открыть PDF в просмотрщике. Показать комиссии:
   - Шапку с датой, временем обработки, версиями моделей.
   - Встроенный график спектра.
   - Таблицы функциональных групп и кандидатов.
   - На второй странице — Grad-CAM-карту (если была включена).

**Что говорить:** *«Отчёт генерируется на сервере через ReportLab,
полностью stateless: фронт передаёт JSON-ответ идентификации,
бэкенд рендерит PDF. Никаких внешних сервисов, всё локально».*

## Сценарий 4 (UC-06). Пакетная обработка

**Время:** ~1.5 минуты.

1. Перейти в **«Пакет»**.
2. Перетащить **три** копии `sample.jdx` в зону загрузки.
3. Нажать **«Запустить»**. Файлы обрабатываются последовательно,
   каждый меняет статус «в очереди → обработка → готово».
4. Когда все три закончат — показать таблицу с результатами и
   нажать **«Экспорт CSV»**.

**Что говорить:** *«Batch-эндпоинт — UC-06, FR-08. Ограничения
50 МБ на файл и 500 МБ на архив зафиксированы в главе 4.11 как
защита от DoS».*

## Сценарий 5 (UC-08). История запросов

**Время:** ~1 минута.

1. Перейти в **«История»**.
2. В таблице видны 4 записи (1 от UC-01 + 3 от UC-06).
3. Кликнуть на запись с этанолом — открывается страница результатов

**Что говорить:** *«История хранит метаданные всех запросов;
полный результат, как видно на экране, доступен в текущем сеансе.
На фазе 2 история будет персистентной, с сохранением спектра и
кандидатов в БД».*

## 6. Резерв (~1.5 минуты): вопросы и ответы

Подготовленные ответы:

- **«Какая точность?»** — *«На фазе 1 (синтетика) метрики не
  достигают целевых значений главы 11. Полный отчёт о
  тестировании — в каталоге `docs/test-report/`, coverage 94%,
  NFR-01 запас ×50. Финальная оценка — после Этапа 20 на
  NIST/SDBS».*
- **«Можно ли запустить локально без интернета?»** — *«Да, после
  однократного `docker compose pull` стек полностью автономен
  (ADR-7). Никаких облачных вызовов».*
- **«Что с производительностью?»** — открыть
  `docs/test-report/benchmark/README.md` и Locust-отчёт.
- **«А GPU нужен?»** — *«CPU достаточно для NFR-01, см. бенчмарк.
  GPU опционален — описан в `docs/INSTALL.md` §10».*

## Fallback-сценарии

### Бэкенд unhealthy после старта
- Открыть терминал: `docker compose logs backend | tail -30`.
- Самое частое — модели в `models/` отсутствуют. Сообщить
  комиссии: *«На демо-стенде синтетические модели, которые могут
  не подгружаться при пересборке. Покажу архитектуру по диаграммам
  главы 4»* и перейти к Swagger-вкладке.

### Сеть медленная / Plotly-график подвисает
- Использовать кнопку **«Использовать образец»** — она загружает
  встроенный спектр без сетевых запросов.

### Браузер не открывается
- Запасной — `curl http://localhost/api/v1/health` в терминале.

### Время заканчивается
- Сценарии 2 и 4 — необязательные. Минимальный путь: UC-01 → UC-05
  → UC-08.

# Сценарий предзащиты (v1.5-predefense)

Этот сценарий **заменяет** разделы выше, когда модели фазы 2 уже
загружены в `models/cnn1d-predefense-0.5.0/` и
`models/contrastive-predefense-0.5.0/`, а БД засеяна реальными
соединениями. Сценарий показывает работу системы на **реальных
NIST-спектрах**, что комиссия принимает как зрелый результат, а не
прототип.

## P0. Подготовка фазы 2 (один раз)

```bash
# В .env (или прямо в docker-compose.yml через override) указать:
# ML_CONTRASTIVE_CHECKPOINT=/models/contrastive-predefense-0.5.0/best.pt
# ML_CNN_CHECKPOINT=/models/cnn1d-predefense-0.5.0/best.pt
# ML_FAISS_ROOT=/models/faiss/contrastive-predefense-0.5.0

python backend/scripts/seed_compounds.py \
    --parquet ml/data/processed/predefense_labeled.parquet

python ml/scripts/build_faiss_index.py \
    --checkpoint models/contrastive-predefense-0.5.0/best.pt \
    --parquet ml/data/processed/predefense_labeled.parquet \
    --phase 2 --data-source real_nist_2000

docker compose down && docker compose up -d --build
```

## P1. Подбор демо-спектров

См. `demo/predefense_spectra/README.md`. Заранее отбирают 5–10
JCAMP-DX из `ml/data/raw/nist/` и копируют на рабочий стол.

## Сценарий 1' (UC-01, чистый случай — этанол)

1. На странице `/identify` загрузить `demo/predefense_spectra/64-17-5.jdx`.
2. После идентификации:
   - top-1 группа: **alcohol_OH** (вероятность близка к 1);
   - top-1 кандидат: **ethanol**, score > 0.9;
   - Grad-CAM ярко горит в районе 3200–3500 см⁻¹.
3. Прокомментировать: *«Это пример, где модель уверена. Реальный
   спектр NIST, обученная фаза 2».*

## Сценарий 2' (UC-01, multi-label — метилбензоат)

1. Загрузить `demo/predefense_spectra/93-58-3.jdx`.
2. Top-3 групп: **ester**, **aromatic_ring**, **ch3_group**.
3. Top-1 кандидат: methyl benzoate.
4. Прокомментировать: *«Модель распознаёт сразу 3 функциональные
   группы. Это multi-label классификация — каждая группа независимо».*

## Сценарий 3' (UC-05, PDF-отчёт)

1. С результатов сценария 2' нажать «Сохранить отчёт».
2. Открыть скачанный `identification-<id>.pdf`. Показать:
   - график спектра 400–4000 см⁻¹;
   - таблицу предсказанных групп с порогами;
   - таблицу кандидатов;
   - вторую страницу с Grad-CAM.

## Сценарий 4' (UC-06, batch)

1. Загрузить 3 файла из `demo/predefense_spectra/` на странице `/batch`.
2. Запустить, дождаться отчёта.
3. Экспортировать CSV.

## Сценарий 5' («трудный» случай)

1. Загрузить заранее выбранный «трудный» спектр (см.
   `demo/predefense_spectra/README.md`).
2. Top-1 кандидат **неверен**, но в top-5 правильное соединение есть.
3. Это **полезный момент** для дискуссии: *«Модель не идеальна, но
   ретривал страхует. На фазе 3 с расширенным датасетом + HP sweep
   планируется поднять top-1 в целевой диапазон главы 6.4.4».*

## Сценарий 6' (UC-08, история)

История теперь содержит ~5 записей с предыдущих сценариев. Показать,
что можно вернуться к каждой.

## Q&A — обновлённые ответы для фазы 2

- **«Какая точность сейчас?»** — заполнить медианой по фактическим
  метрикам из `ml/experiments/predefense/metrics.json`. Ожидаемо
  Macro F1 ~0.5, top-5 ~0.45. *«Это компромисс между размером
  датасета (2000) и сроком одной Kaggle-сессии. Фаза 3 нацелена на
  целевые значения главы 6.4.4».*
- **«Почему не использовали SDBS / Coblentz?»** — *«Они требуют
  ручного скачивания и заняли бы больше недели. NIST WebBook
  полностью автоматизируется через chopralab/candiy_spectrum. Фаза 3
  предусматривает их интеграцию (этап 21 DEVELOPMENT_PLAN.md)».*
- **«Будет ли финальная защита с лучшей моделью?»** — *«Да, после
  предзащиты переходим к фазе 3: extended dataset + hyperparameter
  sweep + полный e2e на финальной модели → тег v2.0-final».*
