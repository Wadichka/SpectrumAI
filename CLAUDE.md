## 3. Архитектура (краткая выжимка из главы 4)

Монолит со слоистой архитектурой, **направленные ациклические зависимости**, шесть слоёв:

1. **Presentation** — SPA на React + Tailwind + Vite, состояние через Pinia, графики Plotly.js, HTTP через Axios.
2. **API Layer** — FastAPI + Pydantic, ASGI Uvicorn, `python-multipart` для аплоада, CORS настроен.
3. **Application Services** — оркестрация: парсинг → предобработка → инференс → ретривал → агрегация.
4. **Domain / ML** — 1D-CNN (multi-label по 20–25 функциональным группам) + двухбашенная схема (спектральная + молекулярная башня) для contrastive retrieval, FAISS как индекс.
5. **Infrastructure** — SQLAlchemy ORM, миграции через Alembic, кэш в Redis, file storage для спектров и моделей.
6. **Persistence** — PostgreSQL **и в dev, и в prod** (поднимается через `docker-compose`). SQLite в памяти допустим **только** для unit-тестов, не зависящих от PostgreSQL-специфики (массивы, JSONB, full-text). Интеграционные и e2e-тесты — на PostgreSQL.

Ключевые модули (4.4):

- `parser` — JCAMP-DX и CSV; валидация структуры и содержимого.
- `preprocessing` — интерполяция на сетку 400–4000 см⁻¹ с шагом 1 см⁻¹, baseline correction, сглаживание, нормирование (SNV/min-max).
- `ml_inference` — 1D-CNN классификатор + спектральная башня (эмбеддинг).
- `vector_search` — FAISS-индекс по молекулярным эмбеддингам референсной библиотеки.
- `interpretation` — Grad-CAM, сопоставление функциональных групп с кандидатами, оценки уверенности.
- `db` — репозитории, метаданные соединений, история запросов.
- `api` — REST-эндпоинты.
- `logging_monitoring` — структурированные логи, метрики.

## 7. Ключевые параметры предметной области

- Диапазон волновых чисел: **400–4000 см⁻¹**, шаг **1 см⁻¹** → вектор длины 3601.
- Multi-label по **20–25 функциональным группам** (точный список — глава 5.4.1). Разметка автоматическая через **RDKit SMARTS-паттерны**.
- Источники датасета: NIST WebBook, SDBS, Coblentz (глава 5.2).
- Поддерживаемые форматы: **JCAMP-DX** и **CSV** (глава 3, FR-01..FR-18).
- Целевые значения метрик — глава 6.4.4 (без согласования не понижать).
- SLA отклика API: < 2 секунд (NFR из главы 3).
- **Отрисовка структурных формул соединений — серверная**. Бэкенд через RDKit генерирует SVG (`Chem.Draw.rdMolDraw2D.MolDraw2DSVG`) и отдаёт его эндпоинтом `GET /api/v1/compounds/{id}/structure.svg`. Фронт просто вставляет полученный SVG. Никаких SmilesDrawer/Kekule на фронте — выкидываем сложную JS-зависимость, RDKit уже в стеке.
- **СУБД — PostgreSQL** для dev и prod (через `docker-compose`). SQLite — только в unit-тестах без специфики PostgreSQL.
- **Кэш — Redis** (отдельный контейнер, подключение через `redis-py` с `aioredis`-совместимым интерфейсом). Не заменять in-memory cache в продакшн-коде, даже «временно».

## 11. Стратегия данных и обучения (трёхфазный подход)

Эта стратегия согласована с автором и **обязательна к соблюдению**. В пояснительной записке (главы 3.6 «Ограничения», 5.1 «Введение в проектирование данных», 6.13 «Выводы») должно быть явно указано, что итоговая система проходит три фазы подготовки данных и обучения.

**Фаза 1 — прототип на синтетических данных (этапы 0–17 плана разработки):**
- Все ML-этапы (4, 5, 6) выполняются на синтетическом датасете из 200–500 искусственно сгенерированных молекул и спектров (сумма гауссов на характеристических частотах). Цель — **отладить пайплайн целиком**, а не достичь целевых метрик.
- На этой фазе допустимо: низкие метрики multi-label F1 (например, 0.3–0.5 на синтетике — нормально), слабый ретривал, нестабильное контрастное обучение. Это не баг, это смысл синтетики.
- Целевые значения метрик из главы 6.4.4 **на этой фазе не достигаются и не должны требоваться**.
- В `MANIFEST.json` для всех моделей фазы 1 ставится поле `"data_source": "synthetic"` и `"phase": 1`.
- Фронтенд, бэкенд, API, Docker, тесты — реализуются полностью и работают с моделями фазы 1.

**Фаза 2 — предзащитный прототип на реальных данных (этапы 18–20 плана разработки):**
- Сжатый сценарий на 1–2 дня работы для **предзащиты**. Цель — получить рабочую систему на реальных спектрах, чтобы продемонстрировать end-to-end на предзащите.
- Сбор ~2000 спектров из NIST WebBook через скрейпер из работы Fine et al. 2020 (см. `download_datasets.py` в корне репо).
- Обучение в одну Kaggle-сессию (~10–14 часов) без hyperparameter sweep.
- Целевые метрики главы 6.4.4 **на этой фазе ещё не достигаются** (ожидаемо F1 macro 0.4–0.6, top-5 retrieval 0.35–0.55). Это компромисс между качеством и сроком.
- В `MANIFEST.json` для моделей фазы 2 ставится поле `"data_source": "real_nist_2000"`, `"phase": 2`.
- Подбирается набор `demo/predefense_spectra/` (5–10 спектров) для уверенной демонстрации на предзащите.

**Фаза 3 — финальное обучение на расширенном датасете (этапы 21–24 плана разработки):**
- Запускается **только после успешной предзащиты** или параллельно с подготовкой к ней.
- Сбор расширенного датасета: NIST WebBook + SDBS + Coblentz, целевая планка 3000–10000 спектров (глава 5.2).
- Обучение в Kaggle Notebooks (T4/P100 GPU) с hyperparameter sweep и сохранением чекпойнтов между сессиями.
- Финальная валидация по протоколу главы 6.5 с целевыми метриками из 6.4.4.
- В `MANIFEST.json` для финальных моделей ставится поле `"data_source": "real_full"`, `"phase": 3`, плюс датасет-метаданные (источники, количество спектров, дата сбора).
- После фазы 3 — обновление пояснительной записки (главы 6 и 11) фактическими метриками.

**Что это означает для Claude Code:**
- Если задача относится к одному из этапов 4–17 (фаза 1) — используй синтетический датасет, не лезь в NIST.
- Если задача относится к этапам 18–20 (фаза 2) — используй предзащитный набор 2000 спектров из NIST через `download_datasets.py`. Hyperparameter sweep здесь **не делать** — цель быстрого обучения.
- Если задача относится к этапам 21+ (фаза 3) — это уже после предзащиты; разрешён hyperparameter sweep, длительное обучение, SDBS и Coblentz, расширенный датасет.
- Не путать фазы: реализация бэкенда не должна ждать «настоящих» моделей, она работает с тем, что есть в `models/` (на фазе 1 — заглушки на синтетике, на фазе 2 — предзащитный прототип, на фазе 3 — финальные модели).
